File Format Documentation

Legacy User Database Files

Introduction

From CodeSnip version 2 to 4 the program stored its "user defined" database in an XML file along with a number of data files located in the user database directory. There was a different user database for each logged on user.

This file format is retained to enable CodeSnip 5 to read databases created by CodeSnip 2 to 4. CodeSnip 5 can load its database from one of these legacy files, but but does not write files in this format.

The master XML file is named database.xml. It contains all the information about user defined snippets and categories except for the source code of each snippet.

The source code for each snippet is stored in separate sequentially numbered .dat data files – one per snippet. Each source code file is referenced by the XML file.

There have been several different versions of the XML file format between CodeSnip 2 and CodeSnip 4. Each of the variations is explained below.

From CodeSnip 5 a new database format was introduced. This format is documented elsewhere.

Encoding

CodeSnip 4 stored all user database files using UTF-8 encoding. Files were saved without any UTF-8 preamble (BOM). The XML processing instruction of database.xml had an "encoding" attribute set to "UTF-8".

Prior to CodeSnip v4 source code data files were encoded using the system default ANSI code page. The XML file was in UTF-8, but its XML processing instruction had no "encoding" attribute.

CodeSnip v5 must be able to read files with all these encoding because it may inherit a copy of a user database in any of them.

File Format

XML File

There were six different versions of the XML file formats – v1 to v6. Nodes from all six versions are explained below with notes describing which versions a node applies to. Where there is no note, the node is valid in all versions.

Notes about how the fields are used in CodeSnip 5 (and later) are displayed in boxes like this one.

XML processing instruction
Attributes:
version
Always set to "1.0"
encoding
Character encoding used for file.
  • versions 1..4: Attribute not provided.
  • version 5 and later: Always set to "UTF-8".
codesnip-data
Parent node for whole file. Attributes are:
watermark
Identifies file as correct type – always set to "531257EA-1EE3-4B0F-8E46-C6E7F7140106".
version
Identifies version of file. Determines which nodes are valid and also the rules concerning content. Valid versions are 1..6.
codesnip-data/categories
Contains list of all categories.
codesnip-data/categories/category
Contains information about a category. Attributes are:
id
Internal (unique) id of category.
Ignored.
codesnip-data/categories/category/description
Description of category.
Added to the set of known Tags.
codesnip-data/categories/category/cat-routines
Contains list of name of all snippets in category. Omitted if there are no snippets in category.
This node, and its children, are ignored.
codesnip-data/categories/category/cat-routines/pascal-name
Contains name of a snippet. One per each snippet in category.
  • versions 1..4: Name must begin with an English language letter or the underscore.
  • version 5 and later: Name can begin with any character that is valid as the first character of a Unicode Pascal identifier.
codesnip-data/routines
Contains a list of all user defined snippets.
codesnip-data/routines/routine
Contains information about a snippet. One per snippet. Attribute:
name
Name of snippet.
  • versions 1..4: Name must begin with an English language letter or the underscore.
  • version 5 and later: Name can begin with any character that is valid as the first character of a Unicode Pascal identifier.
This value is used as the snippet's ID. It may also be used as the value of the snippet's Title property if the codesnip-data/routines/routine/display-name node is missing or empty.
codesnip-data/routines/routine/cat-id
ID of category to which snippet belongs.
The snippet's Tags property is set to contain a single member which is the description of the category with this ID.
codesnip-data/routines/routine/description
Description of snippet.
  • versions 1..5: Content is a single line of plain text.
  • version 6: Content is formatted text encoded in REML mark-up. REML v3 is supported.
This value is parsed and converted to active text and used as the value of the snippet's Description property.
  • versions 1..5: The plain text is parsed as if it was a single paragraph of REML.
  • version 6: the REML is parsed as-is.
codesnip-data/routines/routine/source-code
Name of file containing snippet's source code. No path information included.
The referenced file is read and used as the value of snippet's SourceCode property.
codesnip-export/routines/routine/highlight-source
This node was originally used to determine whether source code should be syntax highlighted, as follows:
  • versions 1..5: The node is not supported and source code should always be syntax highlighted.
  • version 6:
    The node is optional and has a default value of "1" when omitted. Permissible values are:
    • "0" – do not syntax highlight source code.
    • "1" – syntax highlight source code.
This node no longer has any meaning: source code language is now used to determine if or how source code is highlighted. Since this file format has no way of specifying the language directly, this node is used to decide the what language to use and the snippet's LanguageID property is set as follows. If the node is present and has value "0" the language ID is set to "Text". Otherwise the language ID is set to "Pascal".
codesnip-data/routines/routine/display-name
This node was used to specify the snippet's display name, if different to its name, as follows:
  • versions 1..5: Not supported. The display name was always the same as the value of the codesnip-data/routines/routine node.
  • version 6: Display name of snippet. Can contain any characters and need not be unique. Present only if snippet has a display name that is different to the value of the name attribute of the codesnip-data/routines/routine node.
If present and non-empty the value of this node is used for the Title property, otherwise the Title property is set to the value of the codesnip-data/routines/routine node.
codesnip-data/routines/routine/comments
  • version 1: Additional comments about snippets. Optional.
  • version 2 and later: Not supported.
If the codesnip-data/routines/routine/extra node is present or if this node is absent it is ignored. Otherwise the node's text is converted to a single paragraph of active text and used as the first paragraph of the snippet's Notes property.
codesnip-data/routines/routine/credits
  • version 1: Credits for snippets. May contain a single piece of text, delimited by "[" and "]" that can form a hyperlink. URL for the hyperlink is provided in codesnip-data/routines/routine/credits-url.
  • version 2 and later: Not supported.
If the codesnip-data/routines/routine/extra node is present or if this node is absent it is ignored. Otherwise the node's content is parsed, using the URL from the codesnip-data/routines/routine/credits-url node if necessary, into a single active text paragraph this is appended to the snippet's Notes property. Note that the credits paragraph follows after any paragraph obtained from the codesnip-data/routines/routine/comments node.
codesnip-data/routines/routine/credits-url
  • version 1: URL required by codesnip-data/routines/routine/credits node. Present only if codesnip-data/routines/routine/credits requires a hyperlink.
  • version 2 and later: Not supported.
Ignored if the the codesnip-data/routines/routine/extra node is present or if the codesnip-data/routines/routine/credits node is missing or contains no hyperlink placeholder. Otherwise this node is used in the construction of the snippet's Notes property. See codesnip-data/routines/routine/credits above for details.
codesnip-data/routines/routine/extra
  • version 1: Not supported.
  • version 2 and later: Additional information about a snippet. Content is formatted text encoded in REML mark-up.
    • version 2: indicates REML v1
    • version 3: indicates REML v2
    • versions 4..6: indicates REML v3.
When present, the value of this node is parsed and converted to active text and used as the value of the snippet's Notes node. If this node is not present the Notes property will be constructed from the codesnip-data/routines/routine/comments, codesnip-data/routines/routine/credits and codesnip-data/routines/routine/credits-url nodes (see above). If none of those nodes are present then the Notes property is empty.
codesnip-data/routines/routine/standard-format
  • versions 1 and 2: Flag indicating if the snippet is in "standard format". A value of "1" indicates True and "0" indicates False.
  • version 3 and later: Not supported.
If the codesnip-data/routines/routine/kind node is present then this node is ignored. Otherwise it is used to determine the value of the snippet's Kind property, as follows:
  • Value is "1" or the node is not present – the snippet kind is taken to be routine.
  • Value is "0" – the snippet kind is taken to be freeform.
codesnip-data/routines/routine/kind
  • versions 1 and 2: Not supported.
  • version 3 and later: Value indicating kind of snippet. Permissible values are:
    • versions 3 and 4: "freeform", "routine", "type" & "const".
    • version 5 and 6: "freeform", "routine", "type", "const", "class" & "unit".
When present, the value of this node determines the value of the snippet's Kind property. If the node is not present, the property's value determined by the codesnip-data/routines/routine/standard-format node (see above).
codesnip-data/routines/routine/compiler-results
Contains a list of compile results for the snippet.
Used to set the snippet's CompileResults property. Each of the Results for each supported compiler are determined by the codesnip-data/routines/routine/compiler-results/compiler-result child nodes. Any compiler whose results are not listed are et to a query result.
codesnip-data/routines/routine/compiler-results/compiler-result
Entry for each known compiler. Attribute is:
id
Identifies compiler. Valid identifiers are are one of:
  • d2 – Delphi 2 compiler
  • d3 – Delphi 3 compiler
  • d4 – Delphi 4 compiler
  • d5 – Delphi 5 compiler
  • d6 – Delphi 6 compiler
  • d7 – Delphi 7 compiler
  • d2005 – Delphi 2005 compiler
  • d2006 – Delphi 2006 compiler
  • d2007 – Delphi 2007 compiler
  • d2009 – Delphi 2009 compiler
  • d2010 – Delphi 2010 compiler
  • dXE – Delphi XE compiler
  • dXE2 – Delphi XE2 compiler
  • dXE3 – Delphi XE3 compiler
  • dDX4 – Delphi XE4 compiler. Note: This value was named dDX4 in error: it should have been named dXE4 but the erroneous value has been retained for backwards compatibility reasons.
  • dXE5 – Delphi XE5 compiler
  • dXE6 – Delphi XE6 compiler
  • dXE7 – Delphi XE7 compiler
  • dXE8 – Delphi XE8 compiler
  • d10s – Delphi 10 Seattle compiler
  • fpc – Free Pascal compiler
Values are:
  • "Y" – Compiles with the identified compiler.
  • "W" – Compiles with the identified compiler with warnings.
  • "N" – Does not compile with the identified compiler.
  • "Q" – Compile result for this compiler is not known.
Omitting a node for a certain compiler is permitted and is equivalent to specifying "Q".
Each of the child nodes provides the compile result for a single supported compiler. The node values are handled as follows:
  • "Y" or "W" – Compile result set to success.
  • "N" – Compile result set to error.
  • Any other value – Compile result set to query (i.e. unknown).
codesnip-data/routines/routine/units
List of required units.
Used to set the snippet's RequiredModules property. Each of this node's codesnip-data/routines/routine/units/pascal-name child nodes provides the name of a module which is added to the RequiredModules list. If this node is missing, or there are no child nodes, then RequiredModules is an empty list.
codesnip-data/routines/routine/units/pascal-name
Name of a unit within unit list.
codesnip-data/routines/routine/depends
List of required snippets.
Used to set the snippet's RequiredSnippets property. Each of this node's codesnip-data/routines/routine/depends/pascal-name child nodes provides the ID of a snippet which is added to the RequiredSnippets list. If this node is missing, or there are no child nodes, then RequiredSnippets is an empty list.
codesnip-data/routines/routine/depends/pascal-name
Name of a snippet within depends list.
  • versions 1..4: Name must begin with an English language letter or the underscore.
  • version 5 and later: Name can begin with any character that is valid as the first character of a Unicode Pascal identifier.
codesnip-data/routines/routine/xref
List of cross-referenced snippets.
Used to set the snippet's XRefs property. Each of this node's codesnip-data/routines/routine/xref/pascal-name child nodes provides the ID of a snippet which is added to the XRefs list. If this node is missing, or there are no child nodes, then XRefs is an empty list.
codesnip-data/routines/routine/xref/pascal-name
Name of a snippet within cross-reference list.
  • versions 1..4: Name must begin with an English language letter or the underscore.
  • version 5 and later: Name can begin with any character that is valid as the first character of a Unicode Pascal identifier.

Source Code Files

Source code is stored separately from the main XML file. The source code of each snippet has its file. Files are numbered sequentially and have a .dat extension, for example 6.dat.

Source code files are referenced by the codesnip-data/routines/routine/source-code node in the database's XML file.

Summary of Differences Between File Versions

The differences between different user database file versions is summarised below:

Version 1
First version of database.
Version 2
The following nodes are no longer supported:
  • codesnip-data/routines/routine/comments
  • codesnip-data/routines/routine/credits
  • codesnip-data/routines/routine/credits-url
The following nodes was introduced:
  • codesnip-data/routines/routine/extra (uses REML v1 mark-up).
Version 3
The following node is no longer supported:
  • codesnip-data/routines/routine/standard-format
The following node was introduced:
  • codesnip-data/routines/routine/kind
The version of REML supported by the codesnip-data/routines/routine/extra node was updated to v2.
Version 4
The version of REML supported by the codesnip-data/routines/routine/extra node was updated to v3.
Version 5
The XML file's encoding was explicitly set to "UTF-8" by setting the encoding attribute of the XML processing instruction to this value.
Snippet names, wherever they occur in the XML file, can now begin with any character that is a valid first character of a Unicode Pascal identifier. Previously the first character of the attribute had to be one of 'A'..'Z', 'a'..'z' or '_'.
Data files changed to use UTF-8 encoding with no BOM instead of the system default encoding.
New "class" and "unit" snippet kinds supported.
Version 6
A snippet's description is now stored as formatted text using REML mark-up. Previously the description was plain text.
The following node was introduced:
  • codesnip-data/routines/routine/display-name
  • codesnip-export/routines/routine/highlight-source
